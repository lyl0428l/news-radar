阶段一：提示词基础概念

一、什么是提示词（Prompt）？

提示词就是你对AI说的话。

你在对话框里输入的每一句话、每一个要求，都叫提示词。AI根据你给的提示词来理解你要什么，然后生成回答。

打个比方：

  • AI就像一个能力极强但不了解你的新同事
  • 提示词就是你给这个同事下的工作指令
  • 指令越清楚，他干活越靠谱；指令越模糊，结果越离谱

二、为什么同样的问题，写法不同结果差别很大？

看这个对比：

差的提示词

帮我写个文案

问题：AI不知道——写什么文案？给谁看？什么风格？多长？用在哪？

好的提示词

帮我写一段小红书种草文案，推广一款保温杯，
目标用户是25-35岁的女性上班族，
卖点是颜值高、轻便、保温12小时，
风格活泼可爱，300字左右，带emoji

核心区别：好的提示词把"谁、做什么、怎么做、给谁看"说清楚了。

三、提示词的5个基本要素

记住这个公式，写任何提示词都能用：

角色 + 背景 + 任务 + 要求 + 输出格式
要素 说明 示例
角色 让AI扮演谁 "你是一个资深营销专家"
背景 交代前因后果 "我在做一个母婴品牌的推广"
任务 你要AI做什么 "帮我写5条朋友圈文案"
要求 具体的限制条件 "每条不超过100字，风格温馨"
输出格式 结果长什么样 "用编号列表，每条附带一个配图建议"

不是每次都要写满5个，但写得越完整，效果越好。

四、新手最常犯的3个错误

错误1：问得太笼统

❌ "帮我写个方案"
✅ "帮我写一个抖音短视频账号的30天冷启动方案，定位是美食探店，目标涨粉1万"

错误2：一次塞太多事

❌ "帮我写文案，顺便做个表格，再画张图，还有帮我翻译成英文"
✅ 一次只做一件事，做完再提下一个要求

错误3：不会追问和迭代

AI第一次回答不满意是正常的，关键是你要会追问：

→ "这个方向不对，我要的是更偏年轻化的风格"
→ "第三点展开讲讲"
→ "太长了，帮我精简到一半"
→ "换一种写法，参考小红书的口吻"

和AI对话就像和人沟通，反馈越具体，结果越好。

五、Zero-shot vs Few-shot（零样本 vs 少样本）

这两个术语听起来高级，其实很简单：

Zero-shot（零样本）—— 直接让AI做

把下面的评论分类为"正面"或"负面"：
"这家餐厅的菜太难吃了，再也不来了"

不给例子，直接问。简单任务够用。

Few-shot（少样本）—— 先给几个例子再让AI做

请按以下格式对评论进行分类：

评论："味道超棒，下次还来！" → 正面
评论："服务态度很差，等了一小时" → 负面
评论："环境不错但价格偏贵" → 中性

现在请分类：
评论："食材很新鲜，就是位置太偏了"

先给例子让AI学会你要的格式和标准，复杂任务效果更好。

记住：当AI的回答不符合你要的格式或风格时，给它1-3个例子，效果立竿见影。

第一段 | 9:00-10:30 | 什么是LLM

要搞懂的问题

  • LLM = Large Language Model = 大语言模型
  • 它本质上就是一个概率预测器：给它一段话，它预测下一个最可能的词
  • "大"指的是参数量大（几十亿到上万亿个参数）
  • ChatGPT、Claude、DeepSeek、Llama 都是LLM

关键类比

LLM就像一个读了整个互联网的人：
- 它没有"理解"，但它学会了语言的规律
- 它不是在"思考"，而是在"预测最合理的下一个词"
- 它的能力来自于海量数据 + 巨大的模型规模

核心概念速记

概念 一句话解释
参数(Parameters) 模型学到的"知识"，存在数字里。GPT-4据传1.8万亿参数
Token AI处理文字的最小单位，1个汉字≈1-2个token，1个英文单词≈1个token
上下文窗口(Context Window) AI一次能"看到"的文字量。如128K token≈约10万汉字
推理(Inference) 模型生成回答的过程
幻觉(Hallucination) AI一本正经地胡说八道，编造不存在的信息

AI / 机器学习 / 深度学习 / LLM 的关系

人工智能 (AI)
  └── 机器学习 (ML)
        └── 深度学习 (Deep Learning)
              └── 大语言模型 (LLM)

它们是层层递进的包含关系，LLM是目前AI领域最火的一个分支。

神经网络基础（只需理解概念）

输入层 → 隐藏层(很多层) → 输出层

举例：
"今天天气" → [模型内部计算] → "真好"（预测出的下一个词）
  • 神经网络：模仿人脑结构的数学模型
  • 层(Layer)：数据流过的处理阶段，层越多越"深"（所以叫深度学习）
  • 权重(Weight)：每个连接上的数字，训练就是调整这些数字
  • 激活函数：决定神经元是否"激活"的开关

Embedding（词嵌入）—— 非常重要

AI不认识文字，它把文字变成数字向量：

"国王" → [0.2, 0.8, 0.1, ...]
"女王" → [0.3, 0.7, 0.1, ...]
"苹果" → [0.9, 0.1, 0.6, ...]

意思相近的词，向量距离就近
经典公式：国王 - 男 + 女 ≈ 女王

温度(Temperature)

温度低(0.1) → 回答保守、确定、重复性高 → 适合写代码、翻译
温度高(1.0) → 回答多样、有创意、可能跑偏 → 适合创作、头脑风暴

Transformer是几乎所有LLM的底层架构，2017年Google论文《Attention Is All You Need》提出。

为什么Transformer重要？

之前的模型（RNN/LSTM）像是一个字一个字地读书，速度慢，记忆力差。 Transformer能同时看整段文字，还能判断哪些词之间关系最重要。

自注意力机制（Self-Attention）—— 核心中的核心

句子："小明把球传给了小红，因为她站在更好的位置"

AI需要理解 "她" 指的是 "小红" 而不是 "小明"

自注意力机制做的就是：
→ 计算每个词和其他所有词的"关联分数"
→ "她"和"小红"的关联分数最高
→ 所以模型知道"她"="小红"

Transformer的核心组件

输入 → Embedding → 位置编码 → 多头注意力 → 前馈网络 → 输出

简化理解：
1. 把文字变成数字(Embedding)
2. 标记每个词的位置(位置编码)
3. 计算词与词之间的关系(注意力机制)
4. 做进一步的计算和转换(前馈网络)
5. 输出预测的下一个词
组件 做什么
Embedding 文字→数字向量
位置编码 告诉模型词的顺序（第1个、第2个...）
多头注意力 从多个角度分析词之间的关系
前馈网络 对每个位置做独立的非线性变换
Layer Norm 稳定训练过程，防止数值爆炸

GPT vs BERT

GPT（单向）：只看前面的词，预测下一个词 → 擅长生成
  "今天天气" → 预测 "真好"

BERT（双向）：同时看前后文，理解语义 → 擅长理解
  "今天天气[?]好" → 预测 "真"

目前主流的对话AI（ChatGPT、Claude）都是GPT类（自回归模型）。

一个LLM从无到有分三步：

第一步：预训练(Pre-training)
        ↓
第二步：指令微调(SFT / Instruction Tuning)
        ↓
第三步：人类反馈强化学习(RLHF)

第一步：预训练

喂入海量文本（书籍、网页、代码...）
任务：预测下一个词
结果：学会了语言规律，但不会"听话"

就像一个博览群书但没有社会经验的人
  • 数据量：万亿级token
  • 算力需求：数千张GPU，训练数月
  • 成本：数百万到数千万美元

第二步：指令微调（SFT）

用 <指令, 回答> 这样的配对数据来训练

例：
指令："请总结这篇文章"
回答："这篇文章主要讲了..."

结果：模型学会了"听指令办事"

第三步：RLHF（人类反馈强化学习）

1. 给模型一个问题，让它生成多个回答
2. 人类标注员给这些回答排序（哪个好哪个差）
3. 训练一个"奖励模型"学习人类偏好
4. 用强化学习让模型的回答越来越符合人类偏好

结果：回答更安全、更有用、更像人话

这就是ChatGPT比普通GPT好用得多的关键原因。

二、神经网络 —— LLM的"骨架"

什么是神经网络？

先忘掉所有技术术语，看这个比喻：

想象一个工厂的流水线：

原材料(输入) → 工位1 → 工位2 → 工位3 → ... → 成品(输出)

每个工位上有一个工人(神经元)
每个工人做一个简单的操作：
  1. 接收上一个工位传来的半成品
  2. 按自己的"手法"加工一下
  3. 传给下一个工位

"手法" = 权重(weight)
"加工" = 数学计算（乘以权重，加起来）

训练模型 = 不断调整每个工人的"手法"，直到最终产品质量最好

用更具体的例子

假设AI要判断一句话是"正面"还是"负面"：

输入："这家店的菜太好吃了"

第一层（输入层）：
  把每个词变成数字传进去

第二层（隐藏层1）：
  识别出 "好吃" 这个关键词
  识别出 "太...了" 这个加强语气的结构

第三层（隐藏层2）：
  综合判断：正面词 + 加强语气 = 非常正面

第四层（输出层）：
  输出结果：正面 95%，负面 5%

关键概念对照表

概念 类比 作用
神经元 流水线上的一个工人 接收输入、做计算、传出结果
层(Layer) 流水线的一个工位 一组神经元一起工作
权重(Weight) 工人的操作手法 决定输入数据的重要程度
偏置(Bias) 工人的习惯偏好 微调输出结果
激活函数 质检员 决定这个工人的结果要不要传给下一层
深度 流水线有多少道工序 层数越多，能学到的规律越复杂

为什么叫"深度"学习？

浅层网络：输入 → 1-2层 → 输出  （只能学简单规律）
深度网络：输入 → 几十上百层 → 输出（能学复杂规律）

GPT-3 有96层
越深 → 越能理解复杂的语言现象（比如反讽、隐喻、长距离上下文）

三、Embedding（词嵌入）—— AI理解语言的方式

这是最重要的概念之一，很多人卡在这里。

核心问题：AI不认字，怎么办？

人看到 "猫" → 脑子里浮现毛茸茸的小动物
AI看到 "猫" → 看不懂，它只认数字

解决方案：把每个词变成一串数字（向量）

什么是向量？

别被"向量"吓到，就是一组数字：

用3个维度来描述水果：

          甜度  大小  水分
苹果  →  [0.7, 0.5, 0.6]
西瓜  →  [0.8, 0.9, 0.9]
柠檬  →  [0.1, 0.3, 0.7]

苹果和西瓜都甜，所以它们在"甜度"这个维度上数值接近。 柠檬不甜，所以"甜度"维度的数值低。

Embedding做的就是同样的事，只不过维度不是3个，而是几百上千个（GPT用的维度超过12000）。

更直观的例子

假设我们用4个隐藏维度来表示词语：

        皇室  性别  年龄  权力
国王 → [0.9, 0.8, 0.6, 0.9]
女王 → [0.9, 0.2, 0.6, 0.9]
男孩 → [0.1, 0.8, 0.2, 0.1]
女孩 → [0.1, 0.2, 0.2, 0.1]

观察规律：
国王和女王：只有"性别"维度不同
男孩和女孩：只有"性别"维度不同

所以才有经典公式：
国王 - 男(性别维度) + 女(性别维度) ≈ 女王
[0.9,0.8,0.6,0.9] - [0,0.6,0,0] + [0,0,0,0] ≈ [0.9,0.2,0.6,0.9]

Embedding的实际意义

意思相近的词 → 向量距离近（数字差不多）
意思不同的词 → 向量距离远（数字差别大）

"开心" 和 "快乐" → 向量很近  ← AI知道它们是近义词
"开心" 和 "桌子" → 向量很远  ← AI知道它们没关系

重要：这些维度的含义不是人定义的，是模型自己从海量文本中学出来的。

Embedding在哪里用到？

应用 怎么用
LLM输入 每个词先变成向量，才能送进模型计算
语义搜索 把问题和文档都变成向量，找最相似的
RAG 把知识库文档embedding化，存入向量数据库
推荐系统 把用户和商品都embedding化，匹配兴趣

四、Temperature（温度）—— 控制AI的"创造力"

一句话解释

温度控制的是AI在选择下一个词时有多"随机"。

详细解释

AI生成回答时，每一步都在预测"下一个词是什么"，它会给每个候选词打分：

问题："法国的首都是___"

候选词及概率：
  巴黎  → 90%
  里昂  → 5%
  马赛  → 3%
  北京  → 0.1%
  香蕉  → 0.001%

温度决定了AI怎么从这些候选词中选择：

温度 = 0（最保守）
  → 永远选概率最高的"巴黎"
  → 每次回答都一样
  → 适合：事实问答、代码、翻译

温度 = 0.7（适中）
  → 大概率选"巴黎"，偶尔选"里昂"
  → 回答有一定变化
  → 适合：日常对话、写作

温度 = 1.5（很高）
  → 什么词都可能被选中
  → 回答天马行空，可能不靠谱
  → 适合：创意写作、头脑风暴

生活类比

点菜的比喻：

温度低 = 你是个固执的人，每次去餐厅只点招牌菜
温度高 = 你是个冒险家，闭着眼随便指一道菜

温度适中 = 大概率点招牌菜，偶尔尝试新的

实际体验

你可以打开ChatGPT或任何AI工具，试试这个对比：

同一个提示词："写一句关于秋天的诗"

温度0.1（多问几次结果差不多）：
  → "秋风送爽叶飘零，一地金黄染山林"

温度1.2（每次结果都不一样）：
  → "银杏铺了满街的金色信笺，等风来念"
  → "秋天是大自然写给夏天的分手信"
  → "一片叶子坠落，像星球告别轨道"

五、总结串联

把这些概念连起来看整个过程：

你输入："今天天气真好"

第1步 - Token化：
  "今天" "天气" "真" "好" → 4个token

第2步 - Embedding：
  每个token变成一个高维向量
  "今天" → [0.2, 0.5, 0.1, ...]
  "天气" → [0.8, 0.3, 0.7, ...]
  ...

第3步 - 神经网络处理：
  这些向量经过几十层神经网络的计算
  每一层都在提取更深层的语言规律

第4步 - 预测下一个词：
  模型输出候选词的概率分布
  "，" 40%  "啊" 25%  "！" 15% ...

第5步 - Temperature采样：
  根据温度设定，从候选中选一个词

第6步 - 重复第1-5步：
  把选出的词拼接上去，继续预测下一个词
  直到生成完整回答

以上就是第二段所有核心概念的详细拆解。回顾一下你需要记住的：

  1. 神经网络 = 流水线工厂，层层加工数据，训练就是调整每个工人的手法
  2. Embedding = 把文字变成数字向量，意思相近的词距离相近
  3. Temperature = 控制AI回答的随机程度，低→稳定保守，高→随机创意

Transformer 架构深度拆解

一、Transformer之前的世界 —— 为什么需要它？

在Transformer之前，处理语言用的是RNN/LSTM

RNN的工作方式就像"一个字一个字地读书"：

句子："我 今天 去 北京 参加 一个 很 重要 的 会议"

RNN处理过程：
  读"我" → 记住 → 读"今天" → 记住 → 读"去" → 记住 → ...

问题来了：
  读到"会议"的时候，前面的"我"已经记不太清了
  就像你看一本很长的书，看到最后几章忘了开头讲什么

RNN的三大致命缺陷

缺陷 类比
健忘 句子越长，前面的信息丢失越多
太慢 必须一个字一个字按顺序处理，不能并行
长距离依赖差 "小明...（中间隔了100个词）...他"，搞不清"他"是谁

Transformer来了，一次性解决所有问题

Transformer的工作方式就像"一眼看完整页书"：

句子："我 今天 去 北京 参加 一个 很 重要 的 会议"

Transformer处理过程：
  同时看到所有词 → 计算每两个词之间的关系 → 一步到位

优势：
  ✅ 不健忘：所有词同时处理，不存在遗忘
  ✅ 速度快：可以并行计算，GPU跑得飞快
  ✅ 长距离依赖强：句首和句尾的关系也能直接计算

二、自注意力机制（Self-Attention）—— Transformer的灵魂

如果整个LLM只让你理解一个概念，那就是自注意力机制。

核心思想（一句话）

每个词都去"看看"句子里的其他所有词，判断谁跟自己关系最大。

生活比喻：开会

想象你在开一个10人会议，你是其中一个参会者。

普通开会（RNN）：
  你只能听前一个人说的话，然后传给下一个人
  → 信息传着传着就走样了

注意力开会（Transformer）：
  你可以同时听到所有人说的话
  然后你决定：
    - 张总说的跟我的议题关系很大 → 重点关注（注意力高）
    - 李总说的跟我无关 → 随便听听（注意力低）
    - 王总说的有点相关 → 中等关注（注意力中）

每个参会者都同时做这件事 → 这就是"自注意力"

用具体例子走一遍

句子："小猫坐在垫子上，因为它很柔软"

问题："它" 指的是谁？是 "小猫" 还是 "垫子"？

自注意力机制的处理过程：

"它" 去看句子里每个词，计算关联分数：

  "它" ↔ "小猫"   → 关联分数：0.3
  "它" ↔ "坐在"   → 关联分数：0.05
  "它" ↔ "垫子"   → 关联分数：0.5  ← 最高！
  "它" ↔ "柔软"   → 关联分数：0.1
  "它" ↔ "因为"   → 关联分数：0.02

结论："它"和"垫子"关联最强
因为"柔软"更常用来形容"垫子"而不是"小猫"
所以AI理解了："它" = "垫子"

自注意力的三个核心角色：Q、K、V

这是Transformer论文里的核心概念，听起来很难，但用比喻就很好懂：

想象你在图书馆找书：

Q（Query / 查询）= 你的问题
  "我想找一本关于Python编程的书"

K（Key / 键）= 每本书封面上的标签
  书A标签："Python入门"
  书B标签："法国烹饪"
  书C标签："Python高级编程"
  书D标签："量子物理"

V（Value / 值）= 书的实际内容
  书A内容：Python基础语法...
  书B内容：法餐制作方法...
  书C内容：Python设计模式...
  书D内容：量子力学原理...

过程：
1. 用你的问题(Q) 去和每本书的标签(K) 对比
2. 计算匹配分数：
   书A "Python入门" → 匹配度 0.8（很相关）
   书B "法国烹饪"   → 匹配度 0.01（不相关）
   书C "Python高级" → 匹配度 0.6（相关）
   书D "量子物理"   → 匹配度 0.02（不相关）
3. 根据匹配分数，加权获取书的内容(V)：
   最终结果 = 0.8×书A内容 + 0.01×书B内容 + 0.6×书C内容 + 0.02×书D内容
   → 结果主要包含书A和书C的内容

放回到句子里：

句子中的每个词同时扮演三个角色：

"它" 作为 Q（提问者）："我应该关注谁？"
"小猫" 作为 K（被匹配的标签）："我是小猫"
"小猫" 作为 V（提供的信息）：小猫这个词携带的语义信息

每个词都同时生成自己的 Q、K、V
每个词用自己的 Q 去匹配所有词的 K
根据匹配分数，加权获取所有词的 V
→ 最终得到一个融合了上下文信息的新向量

注意力分数可视化

句子："The cat sat on the mat because it was soft"

       The  cat  sat  on  the  mat  because  it  was  soft
  it  [0.05 0.25 0.03 0.02 0.05 0.45  0.03  0.01 0.02 0.09]
       ↑                        ↑
      低关注                  高关注

颜色越深 = 注意力越高
"it" 最关注 "mat"(垫子)，其次关注 "cat"(猫)

三、多头注意力（Multi-Head Attention）—— 多角度观察

为什么一个注意力不够？

句子："小明在银行旁边的河边钓鱼"

"银行" 这个词需要从多个角度理解：
  - 角度1（语义）：这里的"银行"是金融机构还是河岸？
  - 角度2（位置）："银行"和"旁边""河边"是什么空间关系？
  - 角度3（句法）："银行"在句子中做什么成分？

一个注意力头只能关注一个角度
多个头 → 同时从多个角度分析

生活比喻：审稿团队

一篇文章交给审稿团队，每个审稿人关注不同方面：

审稿人1（头1）：关注语法是否正确
审稿人2（头2）：关注逻辑是否通顺
审稿人3（头3）：关注论据是否充分
审稿人4（头4）：关注结构是否合理
...

每个人独立审稿 → 最后把所有人的意见综合起来

这就是多头注意力：
  GPT-3 有 96 个注意力头
  → 相当于96个审稿人同时从不同角度分析句子

简化流程

输入向量
  ↓
┌──────────┬──────────┬──────────┐
│  头1      │  头2      │  头3     │  ...（多个头并行）
│ 关注语义  │ 关注位置  │ 关注句法  │
│ Q1,K1,V1 │ Q2,K2,V2 │ Q3,K3,V3 │
└──────────┴──────────┴──────────┘
         ↓ 合并所有头的结果
    综合后的上下文向量

四、位置编码（Positional Encoding）—— 让AI知道词序

问题

Transformer同时看所有词，但这带来一个新问题：

"猫追狗" 和 "狗追猫" — 词完全一样，意思完全不同！

如果不告诉AI词的顺序，它分不清这两个句子

解决方案：给每个位置加一个"编号"

就像排队取号：

"猫  追  狗"
 #1  #2  #3

"狗  追  猫"
 #1  #2  #3

具体做法：
原始向量     +  位置编码     =  最终输入
"猫"[0.3,0.8] + 位置1[0.1,0.0] = [0.4,0.8]
"追"[0.5,0.2] + 位置2[0.0,0.1] = [0.5,0.3]
"狗"[0.7,0.6] + 位置3[0.1,0.1] = [0.8,0.7]

这样即使同一个词，在不同位置的向量就不一样了
AI就能区分"猫追狗"和"狗追猫"了

类比

就像你寄信：

信件内容 = 词的Embedding向量（"这个词是什么"）
邮政编码 = 位置编码（"这个词在第几个位置"）

没有邮编，信就不知道该送到哪
没有位置编码，AI就不知道词的顺序

五、前馈网络（Feed-Forward Network）—— 深度加工

在注意力层之后

注意力机制做完之后：
  每个词已经知道了"和其他词的关系"

但还不够，需要进一步加工：
  前馈网络 = 对每个词的信息做独立的深度处理

类比：注意力 vs 前馈网络

注意力机制 = 小组讨论
  → 每个人听听别人的意见，获取信息

前馈网络 = 独立思考
  → 每个人根据讨论结果，自己深入思考
  → 形成更深层的理解

先讨论，再思考 → 理解更深
先注意力，再前馈 → 特征更丰富

结构

输入 → 扩展（放大4倍）→ 激活函数 → 压缩（缩回原大小）→ 输出

就像：
  你的笔记（256维）
  → 展开成详细的草稿（1024维）
  → 划掉不重要的部分（激活函数过滤）
  → 整理成精炼的总结（256维）

六、Layer Norm（层归一化）—— 稳定训练

问题

神经网络有几十上百层，数据经过层层计算：
  每层的输出数值范围可能差别很大

第1层输出：[0.2, 0.5, 0.3]
第10层输出：[152.3, -89.7, 203.1]  ← 数值爆炸了！
第50层输出：[999999, -888888, ...]  ← 完全失控

这叫"梯度爆炸"，模型根本没法训练

Layer Norm的作用

就像考试成绩标准化：

原始分数：语文95, 数学30, 英语150（满分不同，没法比较）
标准化后：语文0.8, 数学0.6, 英语0.75（统一尺度，可以比较了）

Layer Norm对每一层的输出做类似的事：
  把数值拉回到一个稳定的范围（均值0，方差1附近）
  → 防止数值爆炸或消失
  → 让训练过程更稳定

七、把所有组件拼起来 —— 完整的Transformer Block

一个Transformer Block（积木块）的完整流程：

输入
  │
  ▼
┌─────────────────────────┐
│   多头自注意力            │  ← "小组讨论"：每个词看看其他词
│   (Multi-Head Attention) │
└────────────┬────────────┘
             │
  残差连接 + Layer Norm       ← "别忘了原始信息" + "稳定数值"
             │
             ▼
┌─────────────────────────┐
│   前馈网络                │  ← "独立思考"：每个词深入加工
│   (Feed-Forward Network) │
└────────────┬────────────┘
             │
  残差连接 + Layer Norm       ← 同上
             │
             ▼
输出

残差连接是什么？

就像考试时的"参考答案"：

没有残差连接：
  你做题 → 只看你的答案

有残差连接：
  你做题 → 你的答案 + 原题信息 合在一起

好处：
  即使中间某一层"算错了"，原始信息还在
  防止深层网络中信息丢失
  
公式理解：输出 = 这一层的处理结果 + 原始输入

GPT-3 = 96个这样的Block堆起来

输入 → Block1 → Block2 → Block3 → ... → Block96 → 输出

每个Block都在做：
  1. 看看上下文关系（注意力）
  2. 深入处理信息（前馈网络）

经过96层处理后，模型对语言的理解足够深

八、GPT vs BERT —— 两种不同的Transformer用法

原始Transformer其实分两部分

原始论文里的Transformer是做翻译的：

编码器(Encoder)            解码器(Decoder)
  ↓                          ↓
理解输入的句子     →      生成翻译后的句子
"我爱你"                   "I love you"

后来大家发现，拆开来用效果更好：

只用编码器 → BERT（擅长理解）
  "今天天气[?]好" → 预测中间的词 → "真"
  应用：搜索引擎、文本分类、情感分析

只用解码器 → GPT（擅长生成）
  "今天天气" → 预测下一个词 → "真好"
  应用：对话、写作、代码生成

现在主流的AI助手（ChatGPT、Claude、DeepSeek）都是GPT路线

核心区别

BERT（双向）：
  "我 [?] 北京"
  可以同时看左边的"我"和右边的"北京"来预测中间的词
  → 理解能力强

GPT（单向）：
  "我 去 北京 ___"
  只能看前面的词来预测下一个词
  → 生成能力强
  
  为什么GPT只看前面？
  因为生成文本时，后面的词还没产生，当然看不到

一张图总结

           Transformer
           /         \
      Encoder       Decoder
         |             |
       BERT          GPT
         |             |
     理解任务       生成任务
     搜索/分类     对话/写作/代码

当你在ChatGPT中输入"写一首关于春天的诗"时：

第1步：分词(Tokenization)

第2步：Embedding

    ├─ 前馈网络：深度加工
    └─ 残差连接 + Layer Norm

第5步：输出预测
  最终向量 → 概率分布
  "春"30% "花"15% "万"10% "阳"8% ...

第6步：Temperature采样
  根据温度选一个词 → "春"

  "写一首关于春天的诗 春风" → 预测下一个词 → "拂"
  ...
  
  直到生成完整的诗

LLM 训练流程深度拆解

先看全局：三步造出一个AI助手

第一步：预训练 (Pre-training)
  → 读遍整个互联网，学会"语言是什么"
  → 产出：一个会说话但不听话的"学者"

第二步：指令微调 (SFT - Supervised Fine-Tuning)
  → 用<指令,回答>的配对数据教它"听话办事"
  → 产出：一个能听懂你要求的"员工"

第三步：人类反馈强化学习 (RLHF)
  → 让人类告诉它哪个回答好、哪个差
  → 产出：一个回答又好又安全的"专业助手"

生活类比：培养一个员工

预训练  = 这个人从小到大读了所有的书，知识渊博
         但你问他问题，他可能给你背一段百科全书，答非所问

SFT    = 入职培训，教他"领导问什么你就答什么，格式要规范"
         他开始能正常回答问题了，但有时候回答质量参差不齐

RLHF   = 上岗后老员工带着他：
         "这个回答好，继续保持"
         "那个回答不行，太生硬了"
         他越来越懂得什么样的回答让人满意

一、预训练（Pre-training）—— 最重要、成本最高的一步

核心任务：预测下一个词

预训练的任务极其简单，就一件事：

给你一段话的前半部分，预测下一个词

训练数据："今天天气真好，我想去公园"

第1次：输入 "今天" → 预测 "天气"
第2次：输入 "今天天气" → 预测 "真"
第3次：输入 "今天天气真" → 预测 "好"
第4次：输入 "今天天气真好，" → 预测 "我"
第5次：输入 "今天天气真好，我" → 预测 "想"
...

就这么简单。但当你用万亿级的数据做这件事，奇迹就出现了。

为什么这么简单的任务能产生"智能"？

想想看，要准确预测下一个词，模型必须学会：

"法国的首都是___"
  → 要预测"巴黎"，必须知道地理知识

"if x > 0: return___"
  → 要预测代码，必须理解编程逻辑

"他虽然很努力，___成绩还是不好"
  → 要预测"但是"，必须理解转折关系

"患者出现发热、咳嗽症状，建议___"
  → 要预测医疗建议，必须有医学知识

为了把"下一个词"预测准确：
  语法、语义、常识、专业知识、逻辑推理、代码能力...
  全部都被隐式地学会了

训练数据：吃什么决定了能力

预训练数据就像人的"饮食"：

吃得好（数据质量高）→ 身体好（模型能力强）
吃得多（数据量大）  → 更壮（模型更全面）
偏食（数据不均衡）  → 某些能力缺失

GPT-3 的训练数据构成（约3000亿token）：
┌──────────────────────────────┐
│  网页爬取（Common Crawl）60%  │  ← 互联网上的各种内容
│  书籍语料              16%   │  ← 出版的书籍
│  维基百科               3%   │  ← 百科知识
│  代码数据              13%   │  ← GitHub等代码仓库
│  其他                   8%   │  ← 新闻、论文等
└──────────────────────────────┘

数据清洗 —— 比想象中重要得多

互联网上的数据大部分是"垃圾"：

原始数据：
  ❌ 广告弹窗文本："立即点击领取优惠！！！"
  ❌ 乱码页面："#@$%^&*"
  ❌ 重复内容：同一篇文章在100个网站上转载
  ❌ 低质量内容："啊啊啊啊啊好帅"
  ❌ 有害内容：暴力、歧视、虚假信息

数据清洗要做的事：
  1. 去重：删除重复内容
  2. 过滤：删除低质量/有害内容
  3. 语言分类：区分中文、英文等
  4. 格式清理：去掉HTML标签、广告等
  5. 质量打分：给每段文本评分，只留高分的

"垃圾进，垃圾出"（Garbage In, Garbage Out）
数据质量直接决定模型质量

训练过程到底在干什么？

简化版本：

1. 拿出一段文本
   "我喜欢吃苹果"

2. 模型预测下一个词
   输入"我喜欢吃" → 模型预测 "香蕉"（预测错了）

3. 计算损失
   正确答案是"苹果"，模型说"香蕉"
   损失 = 预测结果与正确答案的差距

4. 反向传播
   → 从输出层往回看，找到哪些权重导致了错误
   → 微调这些权重，让下次更可能预测出"苹果"

5. 重复以上步骤万亿次
   每次微调一点点 → 积少成多 → 模型越来越准

类比：学骑自行车

你第一次骑自行车：

1. 你尝试骑（前向传播）→ 摔倒了（预测错误）
2. 你感受到"往左偏太多了"（计算损失）
3. 你大脑调整"下次往右偏一点"（反向传播，调整权重）
4. 再试一次 → 还是摔，但稍微好一点
5. 几百次之后 → 你会骑了

模型训练就是这个过程，只不过：
  你的大脑   → 神经网络

Scaling Law（规模定律）—— 大力真的出奇迹

OpenAI发现了一个重要规律：

  模型越大 + 数据越多 + 算力越强 → 效果越好

而且这个关系非常稳定、可预测：

  参数翻10倍 → 效果提升一个档次
  数据翻10倍 → 效果提升一个档次
  算力翻10倍 → 效果提升一个档次

这就是为什么各家公司疯狂堆规模的原因
"大力出奇迹"在LLM领域确实成立（至少目前为止）

但也有极限：
  数据快要被用完了（互联网数据总量有限）
  成本越来越高（训练一次上亿美元）
  收益递减（从80分到90分比从60分到80分难得多）

预训练结束后，模型是什么状态？

预训练完的模型（叫"base model/基座模型"）：

它会的：
  ✅ 续写文本："从前有座山" → "山里有个庙，庙里有个老和尚..."
  ✅ 语言知识：语法、词汇、句式都掌握了
  ✅ 世界知识：知道巴黎是法国首都、水的化学式是H2O
  ✅ 代码能力：能写出语法正确的代码

它不会的：
  ❌ 对话：你问"你好"，它不会回"你好"，而是续写成一篇文章
  ❌ 听指令：你说"总结一下"，它可能接着写"这篇文章的要点..."然后编一堆
  ❌ 安全性：可能会生成有害内容，因为训练数据里什么都有
  ❌ 格式控制：不懂你要列表、表格还是段落

示例：
  你输入："请问中国的首都是哪里？"
  
  基座模型可能回答：
  "请问中国的首都是哪里？这是一个常见的地理问题。
   在百度知道上，很多人会问这样的问题。事实上，
   中国作为一个历史悠久的国家..."（开始写小作文）

  而不是直接回答："北京"

所以需要第二步——指令微调（SFT）来教它"好好说话"

二、指令微调（SFT）—— 教模型"听话"

核心思想

用人工标注的 <指令, 标准回答> 配对数据来训练

让模型学会：
  "别人问什么，你就答什么，别乱发挥"

训练数据长什么样？

数据示例1：
  指令："请用一句话解释什么是光合作用"
  回答："光合作用是植物利用阳光、水和二氧化碳制造氧气和养分的过程。"

数据示例2：
  指令："把下面这段话翻译成英文：今天天气很好"
  回答："The weather is very nice today."

数据示例3：
  指令："写一个Python函数，计算列表中所有数字的平均值"
  回答："
  def average(numbers):
      if not numbers:
          return 0
      return sum(numbers) / len(numbers)
  "

数据示例4：
  指令："我感到很焦虑，怎么办？"
  回答："焦虑是一种正常的情绪反应。以下是一些缓解方法：
        1. 深呼吸练习...
        2. 适当运动...
        3. 如果持续严重，建议寻求专业帮助。"

数据量对比

预训练数据：万亿级token（海量，质量参差不齐）
SFT数据：    几万到几十万条（少量，但每条都是精心标注的）

比喻：
  预训练 = 读了一万本书（量大）
  SFT    = 做了100道精选练习题（量小但有针对性）

一个做过100道高质量练习题的学生
比一个只是死读书的学生，考试成绩好得多

SFT的训练过程

和预训练非常类似，但有关键区别：

预训练：
  输入："今天天气" → 预测下一个词 → "真好"
  （随机文本，无结构）

SFT：
  输入："用户：请解释量子力学\n助手：" → 预测回答 → "量子力学是..."
  （结构化的指令-回答格式）

模型学会的不只是"预测下一个词"
而是"在收到指令后，生成符合要求的回答"

SFT之后的模型

SFT完成后，模型从"学者"变成了"员工"：

之前（基座模型）：
  你："帮我总结这篇文章"
  AI："帮我总结这篇文章。这是许多人在学习中遇到的需求..."（续写）

之后（SFT模型）：
  你："帮我总结这篇文章"  
  AI："这篇文章主要讲了三个要点：1. ... 2. ... 3. ..."（正常回答）

但还有问题：
  ❌ 有时候回答很好，有时候回答很差，质量不稳定
  ❌ 可能会给出不安全、不合适的回答
  ❌ 不够"像人"，回答有时很机械

所以需要第三步——RLHF

三、RLHF（人类反馈强化学习）—— 教模型"回答得好"

核心问题

SFT教会了模型"回答问题"
但没教会它"什么样的回答是好的"

同一个问题可以有很多回答方式：

问题："如何做番茄炒蛋？"

回答A："将番茄切块，鸡蛋打散。先炒鸡蛋盛出，
        再炒番茄，加入鸡蛋翻炒，加盐调味即可。"（简洁实用）

回答B："番茄炒蛋的历史可以追溯到明朝时期。
        番茄学名Solanum lycopersicum..."（答非所问）

回答C："1.准备材料 2.切番茄 3.打鸡蛋 4.热锅..."（很详细，有条理）

哪个好？人类一看就知道，但模型不知道。
RLHF就是让人类来教模型分辨好坏。

RLHF分三步

┌─────────────────────────────────────────┐
│  Step 1：收集人类偏好数据                 │
│    人类标注员对模型的多个回答进行排序      │
│                                          │
│  Step 2：训练奖励模型(Reward Model)       │
│    训练一个模型来模拟人类的偏好判断        │
│                                          │
│  Step 3：用强化学习优化原模型(PPO)         │
│    让模型的回答越来越符合人类偏好          │
└─────────────────────────────────────────┘

Step 1：收集人类偏好数据

给模型一个问题，让它生成多个回答：

问题："推荐一本适合初学者的Python书"

回答A："《Python Crash Course》，适合零基础，
        内容循序渐进，有大量练习项目。"

回答B："Python是一种编程语言，由Guido van Rossum创建..."

回答C："推荐《Python编程：从入门到实践》，
        中文翻译好，讲解清晰。另外也推荐
        《笨办法学Python》作为补充。"

回答D："你为什么想学Python？先想清楚目的再说。"

人类标注员排序：C > A > D > B

意思是：
  C 最好（推荐了多本书，有理由）
  A 不错（推荐了一本，有理由）
  D 一般（反问用户，没直接回答）
  B 最差（答非所问）

Step 2：训练奖励模型

奖励模型 = 一个"评分员AI"

输入：一个问题 + 一个回答
输出：一个分数（越高 = 回答越好）

用Step 1收集的排序数据来训练：
  C得分 > A得分 > D得分 > B得分

训练足够多之后，奖励模型学会了人类的偏好标准：
  ✅ 回答要切题
  ✅ 要有具体信息
  ✅ 要有条理
  ✅ 语气友好
  ❌ 不要答非所问
  ❌ 不要废话连篇
  ❌ 不要有害内容

有了这个"评分员"，就不需要人类一直标注了
奖励模型可以自动给模型的回答打分

Step 3：强化学习优化（PPO）

这一步就像训练宠物狗：

1. 狗（LLM）做一个动作（生成回答）
2. 主人（奖励模型）判断好不好
   好 → 给零食（正向奖励，鼓励这种回答）
   差 → 不给（负向信号，减少这种回答）
3. 狗逐渐学会什么动作能得到零食

具体过程：
1. 给模型一个问题
2. 模型生成回答
3. 奖励模型给回答打分
4. 根据分数调整模型权重：
   高分回答 → 增加生成类似回答的概率
   低分回答 → 降低生成类似回答的概率
5. 重复数十万次

为什么不直接多做SFT，而要用RLHF？

好问题。区别在于：

SFT："照着标准答案学"
  → 只能学到标注员写的那些回答
  → 标注员水平参差不齐
  → 一个问题只有一个"标准答案"

RLHF："学会判断好坏，自己探索更好的回答"
  → 模型可以自己尝试各种回答方式
  → 奖励模型告诉它哪个方向好
  → 模型能找到比标注员写的更好的回答

类比：
  SFT  = 照着答案抄 → 最多和答案一样好
  RLHF = 老师告诉你对错，你自己练 → 可能超越老师

四、DPO —— RLHF的简化替代方案

RLHF很有效，但流程复杂：要训练奖励模型，还要跑PPO

DPO（Direct Preference Optimization，直接偏好优化）：
  跳过奖励模型，直接用人类偏好数据来优化模型

数据格式：
  问题 + 好的回答（chosen） + 差的回答（rejected）

训练目标：
  增加生成"好回答"的概率
  降低生成"差回答"的概率

比喻：
  RLHF = 先培训一个质检员，再让质检员指导工人
  DPO  = 直接拿好产品和差产品给工人看，让他自己领悟

优点：更简单、更稳定、更省资源
现在很多模型（包括Llama、Claude）都在用DPO或其变体

一、API调用 —— 用代码控制AI

什么是API？

API = Application Programming Interface = 应用编程接口

类比：餐厅点菜

直接对话 = 你走进餐厅，坐下来跟服务员面对面说要什么
API调用  = 你打电话给餐厅，说"我要一份宫保鸡丁"，餐厅做好送过来

你不需要知道厨房怎么做菜（模型怎么运行）
你只需要：
  1. 知道餐厅电话号码（API地址）
  2. 有会员卡（API Key）
  3. 说清楚你要什么（发送请求）
  4. 等着收外卖（接收响应）

最简单的API调用示例

# 以OpenAI为例（其他平台大同小异）

from openai import OpenAI

# 1. 拿出会员卡（设置API Key）
client = OpenAI(api_key="sk-你的密钥")

# 2. 打电话点菜（发送请求）
response = client.chat.completions.create(
    model="gpt-4",                    # 选择模型（选哪个厨师做）
    messages=[                         # 对话内容（你要点什么菜）
        {
            "role": "system",          # 系统指令（给厨师的特殊要求）
            "content": "你是一个专业的Python教师，用通俗的语言回答问题"
        },
        {
            "role": "user",            # 用户消息（你的问题）
            "content": "什么是列表推导式？举个例子"
        }
    ],
    temperature=0.7,                   # 温度（口味偏好）
    max_tokens=500                     # 最大回复长度（菜的份量）
)

# 3. 收到外卖（获取回答）
print(response.choices[0].message.content)

messages里的三种角色

messages = [
    {"role": "system",    "content": "..."},  ← 幕后导演：设定AI的人设和规则
    {"role": "user",      "content": "..."},  ← 用户：你说的话
    {"role": "assistant", "content": "..."},  ← AI：模型的回答
]

类比一场戏：
  system    = 导演（告诉演员"你演一个医生"）
  user      = 观众提问
  assistant = 演员回答

多轮对话就是不断往messages列表里追加user和assistant的消息：

messages = [
    {"role": "system",    "content": "你是一个翻译助手"},
    {"role": "user",      "content": "翻译：你好"},
    {"role": "assistant", "content": "Hello"},           ← 第一轮AI回答
    {"role": "user",      "content": "再翻译成日语"},     ← 第二轮用户追问
    {"role": "assistant", "content": "こんにちは"},       ← 第二轮AI回答
    {"role": "user",      "content": "再翻译成韩语"},     ← 第三轮
]

注意：每次调API，你要把历史对话全部传过去
因为API本身是"无记忆"的，它不记得上一次对话

API的关键参数

参数 作用 类比
model 选择用哪个模型 选哪个厨师做菜
messages 对话内容 你的点菜单
temperature 控制随机性(0-2) 口味偏好（保守/冒险）
max_tokens 最大回复长度 菜的份量上限
top_p 另一种控制随机性的方式 从最可能的词中选取比例
stream 是否流式输出 做好一点送一点 vs 全做好再送

流式输出（Streaming）

非流式：
  等3秒 → 一次性收到完整回答
  用户体验：等...等...等...突然出现一大段

流式：
  实时一个字一个字地收到
  用户体验：字在屏幕上逐个蹦出来（就像ChatGPT网页那样）

代码区别：
  stream=False → 等到全部生成完才返回
  stream=True  → 一边生成一边返回，用户体验好

二、RAG（检索增强生成）—— 让AI读懂你的私有资料

核心问题

LLM有两个致命短板：

短板1：知识有截止日期
  "2024年奥斯卡最佳影片是什么？"
  → 如果模型训练数据只到2023年，它答不上来

短板2：不知道你的私有信息
  "我们公司上个月的销售额是多少？"
  → 模型根本没见过你公司的数据

直接把所有资料塞进prompt？
  → 上下文窗口有限（128K token ≈ 10万字）
  → 你的资料可能有几百万字
  → 全塞进去成本也太高了

RAG是什么？

RAG = Retrieval Augmented Generation = 检索增强生成

一句话：先从你的资料库里搜到相关内容，再让AI基于这些内容回答

类比：开卷考试

闭卷考试（普通LLM）：
  只能靠记忆（训练数据）回答，记不住就瞎编

开卷考试（RAG）：
  先翻书找到相关章节（检索）
  再根据书上的内容写答案（生成）
  → 答案又准确又有依据

RAG的完整流程

分两个阶段：

=== 阶段一：建立知识库（提前做好，只做一次） ===

你的资料（PDF/Word/网页/数据库...）
       │
       ▼
  ① 文档切分
       把大文档切成小块（每块200-500字）
       │
       ▼
  ② Embedding
       把每个小块变成向量（还记得第二段讲的吗？）
       "公司Q3销售额增长20%..." → [0.2, 0.8, 0.1, ...]
       │
       ▼
  ③ 存入向量数据库
       所有向量存到专门的数据库里
       （常用的：Pinecone、Milvus、Chroma、FAISS）


=== 阶段二：用户提问时（每次提问都走一遍） ===

用户提问："上个季度销售情况怎么样？"
       │
       ▼
  ④ 问题Embedding
       把问题也变成向量
       "上个季度销售情况" → [0.3, 0.7, 0.2, ...]
       │
       ▼
  ⑤ 向量检索
       在向量数据库里找和问题最相似的文档块
       → 找到："Q3销售报告：本季度销售额增长20%，
              其中华东区贡献最大..."
       │
       ▼
  ⑥ 拼接Prompt
       把搜到的内容和用户问题拼在一起：
       "根据以下资料回答用户问题：
        资料：Q3销售报告...
        问题：上个季度销售情况怎么样？"
       │
       ▼
  ⑦ LLM生成回答
       AI基于你的资料生成回答：
       "上个季度销售额同比增长20%，华东区贡献最大..."

为什么用向量搜索而不是关键词搜索？

关键词搜索的问题：

用户问："员工福利待遇"
资料里写的是："公司为每位同事提供五险一金和补充医疗保险"

关键词搜索：搜"员工福利待遇" → 搜不到！
  因为资料里没有"员工""福利""待遇"这几个字

向量搜索：
  "员工福利待遇" → 向量A
  "公司为每位同事提供五险一金..." → 向量B
  向量A和向量B很相似（意思接近）→ 搜到了！

向量搜索理解的是"语义"，不是"字面"
所以即使用词完全不同，只要意思相近就能搜到

RAG的实际应用场景

场景1：企业知识库问答
  把公司文档、制度、FAQ全部灌进去
  员工直接用自然语言提问
  "请假流程是什么？" → AI从员工手册中找到答案

场景2：客服机器人
  把产品说明书、常见问题灌进去
  用户问"怎么退货" → AI找到退货政策并回答

场景3：论文/报告助手
  上传100篇论文
  "目前关于xxx的研究有哪些主要发现？" → AI从论文中总结

场景4：代码文档助手
  把项目文档、代码注释灌进去
  "这个API怎么调用？" → AI从文档中找到示例

RAG的局限性

❌ 切分不好 → 相关信息被切断了，搜不到完整答案
❌ Embedding质量差 → 搜到的内容不相关
❌ 资料本身有错 → AI基于错误资料回答，还是错的
❌ 问题太模糊 → 搜到的内容不精确
❌ 需要跨文档推理 → 答案分散在多个文档中，拼接困难

所以RAG不是万能的，需要仔细调优：
  文档怎么切？每块多大？
  用什么Embedding模型？
  检索返回几条结果？
  Prompt怎么写才能让AI好好利用检索结果？

三、Agent（智能体）—— 让AI自己使用工具

核心问题

普通LLM的限制：

  ❌ 不能上网搜索最新信息
  ❌ 不能读写文件
  ❌ 不能执行代码
  ❌ 不能发邮件
  ❌ 不能操作数据库
  ❌ 不能调用其他软件

它只能"生成文字"，别的什么都干不了

Agent的解决方案：
  给LLM一套工具，让它自己决定用什么工具、怎么用

Agent是什么？

Agent = LLM（大脑）+ 工具（手脚）+ 规划能力（思考过程）

类比：

普通LLM = 一个坐在轮椅上的天才
  → 什么都知道，但不能动手

Agent = 一个健康的天才
  → 既能想，又能做
  → 可以查资料、写报告、发邮件、跑代码...

Agent的工作流程

用户："帮我查一下北京明天的天气，然后写一条朋友圈文案"

Agent的思考过程（也叫ReAct：Reasoning + Acting）：

思考1："用户要两件事：查天气 + 写文案。先查天气。"
行动1：调用天气API → 获取结果："北京明天晴，25°C，微风"

思考2："天气查到了，现在写朋友圈文案。"
行动2：调用LLM生成文案 → "明天北京25度大晴天，完美的出游日！🌞"

思考3："两件事都做完了，把结果告诉用户。"
最终回答："北京明天晴天，25°C，微风。
          给您写了一条朋友圈文案：
          '明天北京25度大晴天，完美的出游日！🌞'"

Agent能调用哪些工具？

工具类型          举例                    能做什么
─────────────────────────────────────────────────
搜索引擎          Google/Bing搜索         查最新信息
代码执行器        Python解释器            跑代码、做数据分析
文件操作          读写本地文件            处理文档
数据库            SQL查询                 查数据
API调用           天气/地图/翻译API       获取外部服务
浏览器            自动化浏览网页          读取网页内容
邮件              发送/读取邮件           自动化办公
画图              DALL-E/Midjourney       生成图片

Function Calling —— Agent的核心机制

LLM怎么知道有哪些工具、怎么用？

答案：Function Calling（函数调用）

步骤1：你提前告诉模型有哪些工具

  "你有以下工具可以使用：
   - search(query): 搜索互联网
   - calculator(expression): 计算数学表达式
   - send_email(to, subject, body): 发送邮件"

步骤2：用户提问

  "2的100次方是多少？"

步骤3：模型决定调用哪个工具

  模型回复（不是文字，而是一个结构化指令）：
  {
    "tool": "calculator",
    "arguments": {"expression": "2**100"}
  }

步骤4：你的程序执行工具，把结果返回给模型

  计算结果：1267650600228229401496703205376

步骤5：模型根据工具结果生成最终回答

  "2的100次方等于1,267,650,600,228,229,401,496,703,205,376"

整个过程：
  用户 → 模型(决策) → 工具(执行) → 模型(总结) → 用户

类比：你是老板，AI是秘书

没有Agent的AI = 只能动嘴的秘书
  你："帮我订明天去上海的机票"
  AI："好的，你可以打开携程App，搜索北京到上海的航班..."
  （只能告诉你怎么做，不能替你做）

有Agent的AI = 能动手的秘书
  你："帮我订明天去上海的机票"
  AI的思考：
    1. 调用航班查询工具 → 找到3个航班
    2. 根据你的偏好（时间、价格）选最合适的
    3. 调用订票工具 → 下单
    4. 调用邮件工具 → 把行程发到你邮箱
  AI："已为您订好明天8:00的航班，票价680元，行程已发到您邮箱。"

MCP（Model Context Protocol）—— 工具的"统一接口"

问题：每个工具的接口格式都不一样，对接很麻烦

搜索工具的接口格式 ≠ 邮件工具的格式 ≠ 数据库的格式

MCP的解决方案：定义一个统一标准

类比：
  没有MCP = 每个国家的插头不一样，出国要带一堆转换器
  有MCP   = 全世界统一用同一种插头，走到哪都能用

MCP由Anthropic（Claude的公司）提出
目标：让任何工具都能用统一的方式接入任何AI模型
  → 写一次工具适配，所有模型都能用
  → 还在早期，但趋势很明确

四、AI工作流 —— 多个步骤串起来

什么是AI工作流？

单次调用：问一个问题 → 得到一个回答（单步）

类比：
  单次调用 = 手洗一件衣服
  工作流   = 洗衣机的完整程序（浸泡→洗涤→漂洗→脱水→烘干）

五、本地部署 —— 在自己电脑上跑模型

为什么要本地跑？

用API的问题：
  ❌ 要花钱（按token收费）
  ❌ 数据隐私（你的数据发送到了别人的服务器）
  ❌ 依赖网络（没网就不能用）
  ❌ 受限于别人的规则（内容审核、速率限制）

本地部署的优势：
  ✅ 免费（模型开源，跑在自己电脑上）
  ✅ 隐私（数据不出本机）
  ✅ 离线可用
  ✅ 无限制

最简单的本地部署：Ollama

Ollama = 本地跑开源模型的最简单方式

安装（一行命令）：
  Windows：官网下载安装包

量化 —— 让大模型塞进小显卡

还记得第四段提到的"量化"吗？

原始模型：每个参数用16位数字存储（精度高，占空间大）
量化后：  每个参数用4位数字存储（精度略低，省3/4空间）

类比：
  原图（16位）= 高清4K照片，10MB
  压缩图（4位）= 稍微模糊的照片，2.5MB
  → 肉眼差别不大，但省了很多空间

实际效果：
  Llama3 70B 原始版 → 需要140GB显存 → 没人跑得起
  Llama3 70B 4bit量化 → 需要35GB显存 → 一张4090能跑

模型名字里的 Q4、Q8 就是量化级别：
  Q4 = 4bit量化（最省空间，质量损失稍多）
  Q8 = 8bit量化（中间路线）
  F16 = 不量化（效果最好，最费空间）

向量搜索和关键词搜索的区别

好，这个问题非常值得展开讲，它是理解RAG的关键。

向量搜索 vs 关键词搜索 —— 深度拆解

一、先用一个生活场景理解

假设你在一个巨大的图书馆找书，你想找"如何缓解工作压力"相关的书。

=== 关键词搜索（图书管理员A） ===

管理员A的做法：
  在书名和目录里搜索"缓解""工作""压力"这几个字
  
  ✅ 找到了：
    《缓解工作压力的10个方法》     ← 书名直接包含关键词
    《职场压力管理手册》          ← 包含"压力"

  ❌ 找不到：
    《身心放松与冥想指南》        ← 没有"缓解""压力"这几个字
    《高效休息法》               ← 完全不同的用词
    《打工人自救手册》            ← 口语化表达，搜不到

  管理员A只认字，不理解意思


=== 向量搜索（图书管理员B） ===

管理员B的做法：
  理解你想找的"意思"——关于放松、减压、心理健康的内容
  
  ✅ 全都找到了：
    《缓解工作压力的10个方法》     ← 直接相关
    《职场压力管理手册》          ← 直接相关
    《身心放松与冥想指南》        ← 意思相关（放松=缓解压力）
    《高效休息法》               ← 意思相关（休息=减压方式）
    《打工人自救手册》            ← 意思相关（打工人=职场人）

  管理员B理解语义，不拘泥于字面

二、关键词搜索 —— 怎么工作的

原理：字面匹配

关键词搜索的本质就是"找字"

你搜"苹果手机" → 它在所有文档里找包含"苹果"和"手机"的内容

常见技术：
  1. 精确匹配：文档里必须有完全一样的词
  2. 模糊匹配：允许一点点偏差（"手机" 也匹配 "手机壳"）
  3. TF-IDF：统计词频，出现次数多的更相关
  4. BM25：TF-IDF的改进版，目前搜索引擎常用

TF-IDF / BM25 举例

搜索关键词："苹果"

文档A："苹果是一种水果，苹果富含维生素。"
  → "苹果"出现2次，相关性得分高

文档B："我今天吃了一个苹果。"
  → "苹果"出现1次，相关性得分中

文档C："iPhone是苹果公司的产品。"
  → "苹果"出现1次，相关性得分中

问题：你搜的是"苹果"水果，还是"苹果"公司？
  关键词搜索分不清！它只看"苹果"这两个字出现了几次

关键词搜索的优缺点

优点：
  ✅ 速度极快（毫秒级）
  ✅ 技术成熟，实现简单
  ✅ 精确查找特定词非常靠谱（比如搜一个错误代码"Error 404"）
  ✅ 不需要AI模型，成本低

缺点：
  ❌ 只认字不认意思（同义词搜不到）
  ❌ 一词多义分不清（"苹果"是水果还是公司？）
  ❌ 换个说法就搜不到
  ❌ 不理解上下文

三、向量搜索 —— 怎么工作的

原理：语义匹配

向量搜索的本质是"找意思相近的内容"

分两步：
  第一步：把所有文本变成向量（一组数字）
  第二步：计算向量之间的"距离"，距离越近 = 意思越接近

用一个简化例子走完全过程

假设我们的知识库有3段文档：

文档1："员工每月可享受带薪年假5天"
文档2："公司食堂提供免费午餐和晚餐"  
文档3："项目截止日期为下周五"

=== 第一步：全部变成向量（Embedding） ===

文档1 → [0.8, 0.9, 0.1, 0.2]  （福利相关、休假相关）
文档2 → [0.7, 0.3, 0.8, 0.2]  （福利相关、餐饮相关）
文档3 → [0.1, 0.1, 0.2, 0.9]  （工作相关、时间相关）

（实际是几百上千维，这里用4维简化演示）

=== 第二步：用户提问，也变成向量 ===

用户问："公司有什么福利待遇？"
  → [0.8, 0.6, 0.5, 0.1]  （福利相关）

=== 第三步：计算距离 ===

用户问题 vs 文档1（年假）：距离 = 0.2   ← 很近！福利相关
用户问题 vs 文档2（免费餐）：距离 = 0.3  ← 比较近！也是福利
用户问题 vs 文档3（截止日期）：距离 = 0.9 ← 很远！不相关

=== 第四步：返回最相关的结果 ===

结果：文档1（年假）和 文档2（免费餐）

注意：
  用户问的是"福利待遇"
  文档里根本没有"福利待遇"四个字！
  但向量搜索知道"带薪年假""免费午餐"就是"福利待遇"

向量"距离"是什么？

想象一个二维平面（实际是高维空间，道理一样）：

        ↑ 维度2
        |
   "快乐" •    • "开心"     ← 距离近（意思相近）
        |
        |
        |           • "桌子"  ← 距离远（意思不同）
        |
        └──────────────→ 维度1

计算距离的常用方法：

1. 余弦相似度（最常用）
   → 计算两个向量的"夹角"
   → 夹角越小 = 越相似
   → 值范围 0~1，1=完全相同，0=完全无关

2. 欧氏距离
   → 两点之间的直线距离
   → 就是初中学的那个距离公式

日常理解只需要记住：
  数值越接近1（余弦相似度）= 越相关
  数值越接近0 = 越不相关

一秒区分：关键词搜索 vs 向量搜索

最简单的判断方法

关键词搜索 = 找"原话"
向量搜索   = 找"意思"

就这一句话，记住就够了。

用"找人"来理解

假设你要在人群中找一个人，你只知道"那个穿红衣服的高个子"。

=== 关键词搜索的找法 ===

你拿着一张纸条，上面写着"红衣服""高个子"
你走过去一个个看：
  这个人穿红衣服吗？是 ✅ / 不是 ❌
  这个人高吗？是 ✅ / 不是 ❌

只要衣服不是红色 → 跳过（哪怕穿的是深红、酒红也不行）
只要不高 → 跳过

你只认"红衣服"三个字，不认"深红色外套"

你理解了"穿红衣服的高个子"这句话的意思

你认的是"意思"，不是"原话"

  查订单/物流       关键词搜索       有精确的订单号
  问商品信息        向量搜索        用户描述和商品详情用词不同
  问政策/流程       向量搜索        用户用口语，文档用书面语
  报商品名称/SKU   关键词搜索       精确匹配商品编号
  模糊投诉/反馈     向量搜索        需要理解情绪和真实意图

场景三：法律文档检索

业务：律师需要从海量法律条文、案例中找到相关内容

典型问题：
  "关于劳动合同解除的相关法条"
  "类似的侵权案例有哪些"
  "民法典第一百四十三条"

怎么用？

策略：混合搜索，但要分场景加权

=== 场景A：搜具体法条 ===

  用户输入："民法典第一百四十三条"
  
  → 关键词搜索权重 90%
  → 向量搜索权重 10%

  原因：这是精确查找，法条编号必须完全匹配
  "第一百四十三条"和"第一百四十四条"虽然只差一个字
  但内容完全不同，不能搞混


=== 场景B：搜相关法律概念 ===

  用户输入："公司不给加班费怎么办"
  
  → 向量搜索权重 80%
  → 关键词搜索权重 20%

  向量搜索能找到：
    "用人单位应按照劳动合同约定和国家规定，
     向劳动者及时足额支付劳动报酬"
    "劳动者可以向劳动争议仲裁委员会申请仲裁"

  用户说"加班费"，法条里写的是"劳动报酬"
  用户说"怎么办"，法条里写的是"申请仲裁"
  → 只有向量搜索能关联上


=== 场景C：搜类似案例 ===

  用户输入："员工因公受伤，公司拒绝赔偿的案例"
  
  → 向量搜索权重 70%
  → 关键词搜索权重 30%（辅助匹配"工伤""赔偿"等关键词）

  向量搜索的优势：
    "因公受伤" ≈ "工伤事故" ≈ "职业伤害"
    "拒绝赔偿" ≈ "未支付工伤待遇" ≈ "不予理赔"
    → 不同案例的措辞不同，但意思类似的都能搜到

场景四：技术文档 / 代码搜索

业务：程序员在项目文档和代码库中搜索信息

典型问题：
  "NullPointerException在哪里抛出的"
  "用户登录的认证逻辑是怎么实现的"
  "config.yaml里的超时参数"

怎么用？

策略：看搜的是"代码"还是"逻辑"

=== 搜代码/报错/配置 → 关键词为主 ===

  搜"NullPointerException"
    → 关键词精确匹配 → 立刻定位到具体代码行

  搜"config.yaml timeout"
    → 关键词匹配文件名+参数名 → 直接找到

  搜"import pandas"
    → 关键词匹配 → 找到所有用了pandas的文件

  这些都是精确的"字符串"，向量搜索帮不上忙


=== 搜实现逻辑/架构问题 → 向量为主 ===

  搜"用户登录的认证流程是怎么实现的"
    → 向量搜索
    → 找到：auth.py里的login()函数、JWT验证中间件、
           session管理模块...
    → 虽然这些代码/文档里没有"认证流程"四个字
       但语义上都和认证相关

  搜"怎么处理并发冲突"
    → 向量搜索
    → 找到：锁机制文档、乐观锁实现、队列处理...


=== 实际搭配 ===

  开发者搜索系统的理想方案：

  用户输入
       │
       ▼
  ┌─── 判断输入类型 ───┐
  │                     │
  ▼                     ▼
  看起来像代码/报错     看起来像自然语言问题
  (有特殊字符、         (正常的中英文句子)
   驼峰命名、
   文件路径等)
  │                     │
  ▼                     ▼
  关键词搜索为主        向量搜索为主
  (精确找代码)          (理解意图找相关内容)
  │                     │
  └────────┬────────────┘
           ▼
     合并结果返回

场景五：智能客服FAQ

业务：用已有的FAQ问答对来回答用户问题

FAQ库示例：
  Q: "如何修改收货地址？"
  A: "登录APP→我的→订单→修改地址"

  Q: "运费多少钱？"
  A: "满99元免运费，不满99元收取8元运费"

  Q: "支持哪些支付方式？"
  A: "支持微信、支付宝、银行卡支付"

怎么用？

策略：两级搜索

=== 第一级：向量搜索匹配FAQ的问题 ===

  用户问："地址填错了能改吗？"
  
  向量搜索在FAQ的"问题"字段中搜索：
    "地址填错了能改吗" ≈ "如何修改收货地址"
    → 相似度 0.89 ← 高！命中！
  
  直接返回这条FAQ的答案
  → 不需要调用LLM，速度快，成本低


=== 第二级：如果向量搜索置信度不够 → 混合搜索 + LLM ===

  用户问："你们发什么快递？顺丰吗？"

  向量搜索结果：
    "运费多少钱" → 相似度 0.52（有点相关但不确定）
    "支持哪些支付方式" → 相似度 0.31（不太相关）
  
  所有结果置信度都不高（低于0.75）
  → 触发兜底：混合搜索更大的知识库 + 调用LLM生成回答


=== 流程图 ===

  用户提问
       │
       ▼
  向量搜索FAQ库
       │
       ├── 最高相似度 ≥ 0.85 → 直接返回FAQ答案（快+省钱）
       │
       ├── 相似度 0.6~0.85  → 返回答案 + 标注"仅供参考"
       │
       └── 相似度 < 0.6     → 混合搜索知识库 → LLM生成回答
                               （慢一点但更准）

关键技巧：设置"置信度阈值"
  高于阈值 → 直接用搜索结果，省钱省时间
  低于阈值 → 调用LLM，保证回答质量

场景六：多语言搜索

业务：公司有中文和英文文档，用户可能用任意语言提问

知识库：
  文档A（中文）："本产品支持IP68级别防水"
  文档B（英文）："This product is water-resistant up to 50 meters"

怎么用？

=== 关键词搜索 ===

  用户用中文问："防水吗？"
  搜"防水" → ✅ 找到文档A
            → ❌ 找不到文档B（英文里没有"防水"二字）

=== 向量搜索 ===

  用多语言Embedding模型：
  "防水" → 向量A
  "water-resistant" → 向量B
  → 向量A和向量B距离很近（同一个意思）
  → ✅ 中英文文档都能搜到！

向量搜索天然支持跨语言搜索
因为不同语言的"同一个意思"在向量空间里是靠近的

前提：使用支持多语言的Embedding模型
  推荐：BGE-M3、multilingual-e5、Cohere multilingual

总结：实际应用的决策框架

拿到一个搜索需求时，问自己三个问题：

┌─────────────────────────────────────────────────┐
│  问题1：用户输入的是精确信息还是模糊问题？       │
│                                                  │
│    精确（编号/代码/专有名词）→ 关键词为主         │
│    模糊（自然语言/口语）    → 向量为主            │
│    混合（既有精确又有模糊） → 两个都用            │
│                                                  │
│  问题2：知识库的内容和用户的用词差异大吗？         │
│                                                  │
│    差异小（用词一致）      → 关键词就够了          │
│    差异大（同义词多/口语化）→ 必须用向量          │
│                                                  │
│  问题3：对准确率的要求有多高？                    │
│                                                  │
│    不能搜错（法律/医疗/金融）→ 混合搜索+人工复核  │
│    允许模糊（闲聊/推荐）    → 向量搜索足够         │
└─────────────────────────────────────────────────┘

  ──────────────────────────────────────────────────

  法律检索            30%         70%       法条用语和口语差异大

一个完整的混合搜索实现思路

用户提问
     │
     ▼
┌──────────┐
│ 查询预处理 │  提取关键词、实体、判断意图
└─────┬────┘
      │
  ┌───┴───┐
  ▼       ▼
关键词   向量
搜索     搜索
  │       │
  ▼       ▼
结果A    结果B
  │       │
  └───┬───┘
      ▼
┌───────────┐
│ 结果融合    │  去重 + 按加权分数重新排序
│            │  分数 = α×关键词分数 + β×向量分数
│            │  (α + β = 1，根据场景调)
└─────┬─────┘
      ▼
  取Top K结果
      │
      ▼
  拼接到Prompt → 发给LLM → 生成最终回答

核心就三点：

  1. 不是二选一，而是按场景调配比 — 精确查找关键词为主，模糊问题向量为主，拿不准就混合搜索
  2. 先判断用户意图 — 用户输入的是编号/代码还是自然语言，决定了你该侧重哪种搜索
  3. 设置置信度阈值 — 搜索结果分数高就直接用，分数低就兜底调LLM
